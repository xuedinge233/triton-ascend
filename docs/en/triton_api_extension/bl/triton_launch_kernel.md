# triton_launch_kernel

## 1. Overview

`triton_launch_kernel` is a C-language runtime interface exported by the Ascend backend launcher stub shared library (`.so`). It launches a pre-registered Triton kernel onto the NPU after the caller has obtained a valid CANN runtime kernel function handle.

The function is exported with `extern "C"` linkage and is **independent of** the standard Python JIT invocation path (`@triton.jit` → `kernel[grid](...)`). Normal users invoking kernels via `@triton.jit` do not go through this function. It is intended for advanced scenarios requiring C-level kernel launch capability, such as custom deployment pipelines, inference engine integration, and same-signature kernel reuse.

Each JIT compilation artifact (launcher stub `.so`) contains one copy of `triton_launch_kernel` dynamically generated by `generate_npu_wrapper_src()`. The launch argument buffer layout varies across compilation artifacts depending on the kernel signature, metadata (workspace / syncBlockLock / coalesce), device_print, and ffts support. Callers should be aware of this variability (see [Section 8](#8-limitations-and-notes)).

## 2. Function Signature

```c
extern "C" {
void triton_launch_kernel(
    const char* kernelName,
    const void* func,
    rtStream_t stream,
    int gridX,
    int gridY,
    int gridZ,
    const int64_t* shapes_data,
    const int* shape_dims,
    int num_tensors,
    const int* tensor_kinds,
    const void* const* kernel_args,
    const size_t* arg_sizes,
    int num_args
);
}
```

## 3. Parameters

| Parameter | Type | Direction | Description |
|-----------|------|-----------|-------------|
| `kernelName` | `const char*` | in | Kernel name string, used for logging and profiling identification |
| `func` | `const void*` | in | **Kernel function handle registered with CANN runtime.** The caller must have completed `rtDevBinaryRegister` / `rtFunctionRegister` beforehand. Raw `.aocx` file data cannot be passed directly |
| `stream` | `rtStream_t` | in | CANN runtime stream handle on which the kernel will execute |
| `gridX` | `int` | in | Launch grid X dimension. Total block count = `gridX × gridY × gridZ` |
| `gridY` | `int` | in | Launch grid Y dimension |
| `gridZ` | `int` | in | Launch grid Z dimension |
| `shapes_data` | `const int64_t*` | in | Flattened 1-D array of all tensor shapes concatenated in order. Length = `sum(shape_dims[i]) for i in 0..num_tensors-1`. Primarily used for msprof tensor info reporting; not required for kernel execution. May be `nullptr` (in which case `shape_dims` should also be `nullptr`) |
| `shape_dims` | `const int*` | in | Number of dimensions per tensor. Array length = `num_tensors`. Must be `nullptr` if `shapes_data` is `nullptr` |
| `num_tensors` | `int` | in | Total number of tensors, determining the effective length of both `shape_dims` and `tensor_kinds` arrays |
| `tensor_kinds` | `const int*` | in | Tensor kind markers. Array length = `num_tensors`. Values: `0` = INPUT, `1` = OUTPUT, `2` = INPUT_OUTPUT. Primarily used for msprof tensor classification. If `nullptr`, all tensors are treated as INPUT |
| `kernel_args` | `const void* const*` | in | Array of pointers to kernel arguments, in kernel function signature order. Array length = `num_args`. May be `nullptr` if `num_args` is 0 |
| `arg_sizes` | `const size_t*` | in | Size (in bytes) of each kernel argument, one-to-one with `kernel_args`. Array length = `num_args`. May be `nullptr` if `num_args` is 0 |
| `num_args` | `int` | in | Number of kernel arguments. If `num_args > 0` but `kernel_args` or `arg_sizes` is `nullptr`, the function returns silently without launching the kernel |

## 4. Memory Layout and Lifetime

### 4.1 Deep Copy of kernel_args

The function iterates over `kernel_args` and `arg_sizes`, copying each argument value via `memcpy` into a local `std::vector<std::vector<char>> copied_kernel_args` container. Therefore:

- **The caller may free the memory pointed to by `kernel_args` entries immediately after the function returns** without affecting the launch
- The deep copy incurs temporary memory overhead equal to `sum(arg_sizes[i])`

### 4.2 launch_args Contiguous Buffer Layout

The function assembles all launch arguments into a contiguous `std::vector<char> launch_args` buffer with the following layout (each slot aligned per its alignment requirement):

```
[ffts_addr] → [syncBlockLock_ptr] → [workspace_addr_ptr] →
[kernel_arg_0] [kernel_arg_1] ... [kernel_arg_N-1] →
[gridX] [gridY] [gridZ] →
[DTData]   // present only when TRITON_DEVICE_PRINT="true"
```

> **Note:** This layout is dynamically generated by `generate_npu_wrapper_src()` based on the kernel's signature, metadata, and compilation options. Different compilation artifacts may have different layouts.

### 4.3 Workspace and syncBlockLock

- **Workspace**: When `metadata.workspace_size > 0`, workspace memory is allocated internally via `allocate_memory` and managed by `std::shared_ptr` with automatic release when the function returns
- **syncBlockLock**: When `metadata.lock_num > 0`, sync block lock memory is allocated internally via `allocate_sync_block_lock`, initialized to `metadata.lock_init_value`, and managed by `std::shared_ptr`
- On allocation failure, an error message is printed to stderr and the function returns; no exception is thrown

### 4.4 Stream Synchronization Strategy

Depends on the `TRITON_ENABLE_TASKQUEUE` environment variable (default `"true"`):

| Mode | Behavior | When the function returns |
|------|----------|---------------------------|
| TaskQueue enabled (default) | Wraps `rtKernelLaunch` in `std::function<rtError_t()>` and submits it via `triton_async_launch` | Immediately after submission; does not wait for kernel completion |
| TaskQueue disabled | Executes `rtKernelLaunch` synchronously, then calls `rtStreamSynchronize(stream)` | After kernel execution completes |

## 5. Invocation Paths and Use Cases

### 5.1 Path A: Standard Python JIT (Normal Users)

This path does **not** go through `triton_launch_kernel`. It is the standard Triton invocation flow:

```
User code:   kernel[grid](args...)
  │
  ▼
NPULauncher.__call__()                    [driver.py:136]
  │  Dynamically loads .so launcher stub
  │  Calls self.launch(gridX, gridY, gridZ, stream, func, ...)
  ▼
Python C extension launch()              [driver.py:1063 — generated]
  │  PyArg_ParseTuple to parse Python args
  │  Extracts tensorShapes, tensorKinds
  │  Calls _launch(kernelName, func, stream, gridX, gridY, gridZ, ...)
  ▼
_launch() internal function              [driver.py:946 — generated]
  │  Handles workspace / syncBlockLock allocation
  │  Assembles launch_args buffer, calls rtKernelLaunch()
  ▼
rtKernelLaunch() → NPU hardware execution
```

### 5.2 Path B: External C Interface (Directly calling triton_launch_kernel)

For scenarios requiring C-level kernel launch:

```
Third-party C/C++ code
  │  dlopen / dlsym to obtain triton_launch_kernel symbol
  │  or directly link against launcher stub .so
  ▼
triton_launch_kernel()                   [driver.py:800 — generated]
  │  Deep-copies kernel_args
  │  Handles workspace / syncBlockLock allocation
  │  Assembles launch_args buffer, calls rtKernelLaunch()
  ▼
rtKernelLaunch() → NPU hardware execution
```

### 5.3 Use Cases

| Scenario | Recommended Path | Notes |
|----------|------------------|-------|
| Daily Triton operator development | Path A | Use standard `@triton.jit` workflow |
| Custom deployment pipeline | Path B | Pre-registered kernel handle, bypass JIT frontend |
| Inference engine integration | Path B | Engine requires C-level kernel launch interface |
| Same-signature kernel reuse | Path B | Reuse one launcher stub for different compiled kernels |

## 6. Minimal Invocation Examples

### 6.1 C/C++ Example (with Existing func Handle)

The following example assumes the caller has already registered the kernel binary via CANN runtime and holds a valid `func` handle. It demonstrates only the parameter organization for calling `triton_launch_kernel`. The kernel registration flow (`rtDevBinaryRegister` / `rtFunctionRegister`) is standard CANN runtime procedure and is outside the scope of this example.

```c
#include <cstring>
#include <dlfcn.h>
#include <vector>
#include "rt.h"  // CANN runtime header

// Function pointer typedef matching the launcher stub export signature
typedef void (*triton_launch_kernel_t)(
    const char* kernelName,
    const void* func,
    rtStream_t stream,
    int gridX, int gridY, int gridZ,
    const int64_t* shapes_data,
    const int* shape_dims,
    int num_tensors,
    const int* tensor_kinds,
    const void* const* kernel_args,
    const size_t* arg_sizes,
    int num_args
);

void launch_kernel_via_stub(
    const char* stub_so_path,
    const char* kernel_name,
    const void* func,          // kernel function handle registered via CANN runtime
    rtStream_t stream,
    int grid_x, int grid_y, int grid_z)
{
    // 1. Load launcher stub and obtain triton_launch_kernel symbol
    void* handle = dlopen(stub_so_path, RTLD_LAZY);
    if (!handle) {
        fprintf(stderr, "dlopen failed: %s\n", dlerror());
        return;
    }
    auto launch_fn = (triton_launch_kernel_t)dlsym(handle, "triton_launch_kernel");
    if (!launch_fn) {
        fprintf(stderr, "dlsym failed: %s\n", dlerror());
        dlclose(handle);
        return;
    }

    // 2. Assemble kernel arguments
    float alpha = 1.0f;
    int N = 1024;
    const void* arg_ptrs[] = { &alpha, &N };
    const size_t arg_sizes[] = { sizeof(float), sizeof(int) };

    // 3. shapes_data / tensor_kinds (for profiling only; may be empty)
    //    Example with 2 tensors: input shape [1, 1024], output shape [1, 1024]
    const int64_t shapes[] = {1, 1024, 1, 1024};
    const int dims[] = {2, 2};
    const int kinds[] = {0, 1};  // INPUT, OUTPUT

    // 4. Invoke triton_launch_kernel
    launch_fn(
        kernel_name, func, stream,
        grid_x, grid_y, grid_z,
        shapes, dims, 2, kinds,
        arg_ptrs, arg_sizes, 2
    );

    // 5. In non-TaskQueue mode, synchronization can be done here.
    //    TaskQueue mode is enabled by default — the function has already
    //    submitted the task and returned; synchronization is caller-managed.
    dlclose(handle);
}
```

> **Note:** The `func` parameter in this example is a kernel function handle registered via CANN runtime. The registration flow (`rtDevBinaryRegister` / `rtFunctionRegister`) is standard CANN runtime procedure — consult the CANN header (`runtime/kernel.h`) for exact API signatures. The `loadKernelBinary` / `registerKernel` functions in `npu_utils.cpp` provide Python-side wrappers. Whether these helper functions may be documented as public API is a pending decision — see [Pending Items](#pending-items).

### 6.2 Python ctypes Example

Invoking `triton_launch_kernel` from Python via ctypes:

```python
import ctypes
from pathlib import Path

# launcher stub .so path (obtainable via NPULauncher.get_launcher_so_path())
stub_path = Path("/path/to/launcher_cxx11abi1.cpython-39-aarch64-linux-gnu.so")

lib = ctypes.CDLL(str(stub_path))

# Define function signature
lib.triton_launch_kernel.argtypes = [
    ctypes.c_char_p,                # kernelName
    ctypes.c_void_p,                # func
    ctypes.c_void_p,                # stream (rtStream_t)
    ctypes.c_int,                   # gridX
    ctypes.c_int,                   # gridY
    ctypes.c_int,                   # gridZ
    ctypes.POINTER(ctypes.c_int64), # shapes_data
    ctypes.POINTER(ctypes.c_int),   # shape_dims
    ctypes.c_int,                   # num_tensors
    ctypes.POINTER(ctypes.c_int),   # tensor_kinds
    ctypes.POINTER(ctypes.c_void_p),# kernel_args
    ctypes.POINTER(ctypes.c_size_t),# arg_sizes
    ctypes.c_int,                   # num_args
]
lib.triton_launch_kernel.restype = None  # void

# Assemble arguments (func handle acquisition omitted here;
# must be done via CANN runtime Python bindings in practice)
# ...
# lib.triton_launch_kernel(kernel_name, func, stream, ...)
```

### 6.3 Standard Python JIT Invocation (Reference)

Most users do not need to call `triton_launch_kernel` directly. Use the standard `@triton.jit` decorator:

```python
import triton
import torch

@triton.jit
def add_kernel(x_ptr, y_ptr, output_ptr, n_elements, BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(axis=0)
    block_start = pid * BLOCK_SIZE
    offsets = block_start + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n_elements
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)
    output = x + y
    tl.store(output_ptr + offsets, output, mask=mask)

x = torch.randn(1024, device='npu')
y = torch.randn(1024, device='npu')
output = torch.empty_like(x)
grid = lambda meta: (triton.cdiv(1024, meta['BLOCK_SIZE']),)
add_kernel[grid](x, y, output, 1024, BLOCK_SIZE=256)
# This invocation goes through Path A: NPULauncher → launch() → _launch() → rtKernelLaunch()
# It does NOT go through triton_launch_kernel()
```

## 7. Environment Variables

The following environment variables affect the runtime behavior of `triton_launch_kernel`. Default values are verified against the `release/3.2.2` branch source code.

| Variable | Default | Source | Effect |
|----------|--------|--------|--------|
| `TRITON_COMPILE_ONLY` | `"false"` | `driver.py:109` | When `"true"`, skips kernel launch; compile only |
| `TRITON_DEVICE_PRINT` | `"false"` | `driver.py:452-453` | When `"true"`, enables device printf, affecting DTData slot presence in launch_args and CANN DebugTunnel |
| `TRITON_ENABLE_TASKQUEUE` | `"true"` | `driver.py:454-455` | When `"true"`, enables asynchronous TaskQueue mode — the function returns immediately after submission; when `"false"`, waits synchronously for kernel completion |
| `TRITON_GRID_WARN_PRINT` | `"false"` | `driver.py:456-457` | When `"true"`, prints a performance warning to stderr if `blockNum` exceeds the physical core count |

> **Note:** These defaults are verified on the `release/3.2.2` branch only. Other branches or versions may have different defaults; always consult the actual source code.

## 8. Limitations and Notes

### Platform

- Ascend NPU only (Atlas A2/A3 series). GPU is not supported.

### Return Value and Error Handling

- The function returns `void` — **no stable error code contract is provided**
- Parameter validation failures cause silent return (e.g., `num_args > 0` with `kernel_args == nullptr`)
- Resource allocation failures print an error to stderr and return
- The return code of `rtKernelLaunch` is consumed internally and not propagated to the caller
- To confirm kernel execution status, callers should:
  - Non-TaskQueue mode: kernel has completed by the time the function returns; confirmation is implicit
  - TaskQueue mode (default): track task queue status independently

### ABI Stability

- `triton_launch_kernel` is dynamically generated by `generate_npu_wrapper_src()` per kernel compilation. The launch argument buffer layout may differ across launcher stub `.so` files
- Layout-affecting factors include: kernel signature, `metadata.workspace_size`, `metadata.lock_num`, `metadata.coalesce_factor`, `metadata.coalesce_axis`, `metadata.force_simt_only`, `TRITON_DEVICE_PRINT`, and ffts support status
- **`triton_launch_kernel` from different compilation artifacts are not interchangeable.** Callers must ensure the launcher stub `.so` matches the target kernel compilation artifact

### Grid Semantics

- The backend computes `blockNum = gridX × gridY × gridZ` as the total block count for `rtKernelLaunch`
- All three grid values are written to the grid slots in the kernel argument area; the kernel can read them at the corresponding offsets
- When `coalesce_factor > 1` and `coalesce_axis` is valid (0/1/2), the corresponding grid dimension is divided by the coalesce factor before launch (e.g., `gridX = gridX / H`)
- If `auto_map_parallel_blocks` is enabled, `blockNum` is clamped to `min(blockNum, physical_core_count)`
- When `blockNum` exceeds the physical core count and `TRITON_GRID_WARN_PRINT="true"`, a performance warning is printed to stderr

### Miscellaneous

- `kernel_args` deep copy incurs temporary memory overhead of `sum(arg_sizes[i])` bytes
- `shapes_data` / `shape_dims` / `tensor_kinds` are primarily used for msprof tensor info reporting. Omitting them (passing `nullptr`) does not affect kernel correctness, but tensor information will be absent in the profiler
- The generated code depends on the CANN runtime version; the launcher stub must be recompiled after upgrading CANN
